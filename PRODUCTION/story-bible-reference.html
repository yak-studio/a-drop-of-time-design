Canonical narrative design lives in DOCS/ only — no duplicate bible in the repo.

Canonical location

DOCS/ is the source of truth (HTML + styles.css + nav.js). Content was merged from production bible v4.4 narrative on 2026-04-10 into structured pages (lore, plot, act beats, etc.).

What was not cloned verbatim: The old interactive bible’s implementation (modals, every P01-style beat ID, sidebar storyboard chrome) was not copied file-for-file. Design locks and beats were carried into DOCS. If you need legacy beat IDs back, add them beside markers in section breakdown or sprint docs — don’t resurrect a second bible file.

GitHub Pages (this repo)

The workflow .github/workflows/docs-github-pages.yml deploys the DOCS folder whenever you push to main or master (and on manual “Run workflow”).

  1. Create a GitHub repo and push this project.
  2. Repo Settings → Pages → Build and deployment → Source: GitHub Actions.
  3. Push a change under DOCS/ (or run the workflow manually). The site URL will look like https://<user>.github.io/<repo>/ — index.html redirects to MASTER.html.

Can an LLM push for you?

Desktop (Cursor / agent with git): Yes, if the tool session has permission to run git commit / git push and your credentials/SSH are set up. That’s “this environment pushing,” not magic from the model alone.

Phone: Typical chat LLM apps cannot push to your GitHub repo by themselves. You’d use the GitHub app / mobile web (github.dev) / Working Copy (iOS) to commit, or copy files the LLM wrote and paste or upload them — exactly as you said: get the new files, replace, push yourself — that always works.
